Sprint One Planning meeting & setting up Git Hub

Wednesday 11th October

On Wednesday, 11th October the group had a sprint meeting with the product owner. The meeting involved the planning the execution of the first sprint. The product owner made some requirements and tasks they would like in the solution. The team spoke to the product owner and identified the tasks that would take more effort than other tasks and tasks that may not be as essential as others. As the scrum leader, I had taken the leading parts of the conversation and expanded on tasks by asking how they would like specific tasks to be displayed to the user and how they would like them executed.

Following the meeting, our group strategically identified interdependent tasks and prioritized them. We then assigned tasks based on each group member's strengths and weaknesses. To maintain efficiency, we opted to tackle one task at a time, allowing members to move on to the next one independently once completed. My assigned task involved normalizing the existing database, transforming the platform from static data to one with foreign keys.

Then, another group member and I made and tested a git hub repository for the solution, as it will ease the process of group members uploading code at multiple points. Each group member has their own branch that they could make changes to, however, if they wanted to upload changes to the main solution, then it will need to be approved by another team member to ensure that no changes made will add errors to the main version.

Search game by title functionality

Thursday 19th October

I finished the search function and made it so a user can type a game title and the website shows all games with a smiler title. I first had to make a new method in the Game controller. The method uses the HTTPGet to find what the user has typed by using the URL. It looks for 'Title =' with what the user typed to determine what it is looking for, then I had to make a new method in the SQL game repository. The method made in the SQL game repository uses the entity frame work method '.contain' line to compare the title that the user with every game title. Then it takes the games that contain what the user typed after I had to inherit using Encapsulation the method in the interface game repository so the game controller can inherit the method and use it.

To allow the front end to use the API, I first had to make a form in which a user can enter the game's title. The form uses the POST method so the user input is stored in the URL when the page refreshes. The I needed to make a JavaScript file that uses a built-in function to search in the URL for the title to then add the title to the fetch that the JavaScript uses to ask the API for all games that have that title.

Originally, in the SQL game repository, I was going to use the entity framework method '.where' rather than '.contains'; however, if this was implanted instead, it would mean that the user would need to type a name exactly to find the game they are looking for, and this would not be user friendly.

Display the average score against each game

Saturday 28th October

When planning to work on displaying the average score against each game, I was going to make a method that could be used in the SQL game repository, however due to how the get game by title method works, this would not be possible, so I opted for changing each current name method to get data on a game.

I first had to remove the score entity from the game table in the database, as the score for each game will be determined by each review for that game. I then added a score element to the review table. The score element will store a number that a user gives a game, the number will be between one and 100. I then had to add an average score class into the gameDTO so the SQL Game repository would be able to inherit the average score and will be able to store an average score for later use. I then had to edit every method in the SQL game repository that involved the user seeing a game's average score.

Each method had to inherit the get review method in the interface review repository. The get reviews method was used in conjunction with a '.Where' and a 'for each' statement to select reviews that have the same game ID as a game's ID. Each method changed had an added that would add each review's score then divide by how many reviews that fit the criteria, this would find the average score for that game. The methods then use Math.Round() to make the score more user-friendly. In JavaScript files the variable that was used to get a games score was changed to game.averageScore rather than game. Score. The score table header in the UI got changed to Average score.
